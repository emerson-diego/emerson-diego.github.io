How I Build Software with AI in 2026: My Practical Framework for "Vibe Coding" and Harness Engineering

April 17, 2026

Recently, I’ve dived headfirst into the so-called "vibe coding," and reality hit me hard: the execution barrier has plummeted. Today, it’s perfectly feasible to build a clone of a complex system in a day or two using tools like Claude Code (which, by the way, has absolutely dominated the market and become the number one tool among engineers). The code itself has become a commodity.

However, this brutal speed has created a new chasm in our profession: the divide between the "Shippers" (those focused solely on delivering fast) and the "Builders" (those focused on quality and architecture). Shippers are loving the speed, but they are generating mountains of "AI slop" (low-quality, AI-generated code) and technical debt in record time.

Because I consider myself a Builder, I realized that the mental model of treating AI like an "intern"—reading every single line of code it generates—simply breaks down in practice. It doesn't scale.

To avoid drowning in bad code, I fundamentally changed my mental model. I stopped focusing just on the prompt and started designing a control system. Below, I break down exactly how I am running my projects today, blending my personal development touch with the practices of Harness Engineering and Specs-Driven Development.

Step 1: The Foundation (I never start with code)

Before even kicking off the project, my golden rule is to study the product's architecture. AI agents operating without clear boundaries will try to guess the path, and that inevitably leads to hallucinations.

  • Living Documentation: I continuously use the project's documentation to guide the entire workflow. This documentation acts as my single source of truth.
  • The "Spec Pack": In traditional software engineering, we don't push anything to production without clear requirements. I do the exact same thing with AI. For every feature, I create a specification package containing the Product Intent (metrics and constraints) and the Low-Level Design, detailing the acceptance criteria and examples of API contracts or events. If the AI doesn't know exactly what is expected of it, my control system will have nothing to validate against.

Step 2: Collaborative Planning (My secret in the Prompt)

I abandoned the linear working model and adopted a spiral development cycle, focusing strictly on feature by feature. But the real game-changer happens in the exact moment before I tell the AI to implement an issue.

  • The Issue Plan: I never just drop in and say "create endpoint X." When crafting my prompts, I always put together an explicit plan stating: what needs to be done, how it should be done, what the constraints are, and how the result will be validated.
  • Room for AI: After defining these rules, I make room for the AI to analyze the plan and offer its own architectural suggestions. This collaborative planning leverages the model's inferential capacity to anticipate problems before we burn tokens and time actually generating the code.
  • Feedforward Guides: Alongside the plan, I provide instructions that structurally guide the AI before it acts (project conventions, base scripts, etc.). This absurdly increases the chances of it getting things right on the very first try.

Step 3: Implementation and Fast Sensors (Keep Quality Left)

With the plan validated, the AI agent goes to work. However, my confidence doesn't stem from the fact that the AI is "smart," but rather from my network of automated Feedback Sensors.

I strictly follow the "Keep quality left" principle—meaning tests must run as early as possible in the cycle. To ensure my agent's repetition loop is incredibly fast, I rely on purely computational sensors.

As soon as the AI generates the code, it instantly hits linting tools, static type checking, and a rigorous battery of unit and integration tests. These sensors run in milliseconds or seconds. If the AI breaks a contract established in my Spec Pack from Step 1, it catches the error and attempts to correct itself automatically.

Step 4: The "Steering Loop" (How I handle AI failures)

Let's be clear: AI fails, and it fails often. When the error is simple, it reads the test log and fixes it on its own. But when the agent gets stuck in a loop, makes low-confidence decisions, or spits out the dreaded "AI slop," the problem escalates to me.

This is where I step in as a true Harness Engineer. Instead of going in, opening the file, and just patching the line of code the AI messed up, I close the Steering Loop.

  • If the AI didn't know how to implement something, I improve the examples in my specifications documentation.
  • If the error slipped past my tests, I build a new sensor (a new automated test).
  • If it used the wrong pattern, I add a new architectural rule to my Feedforward guides.

I continuously evolve the control system so that this specific error never depends on my manual intervention again in the future.

Step 5: The Delivery Spiral and Customer Feedback

Because I work in a spiral, every time an issue successfully passes through my network of automated sensors, I take the result straight to the client for rapid feedback.

If the client requests business adjustments, the feature loops right back into my collaborative planning phase.

Only when the code is approved, functionally validated, and integrated do I trigger the slow and expensive sensors—like Load and Performance Tests—directly in the CI/CD pipeline. Running heavy tests constantly would completely ruin the project's cost-efficiency and speed.

The Bottom Line

I stopped being an AI-generated code reviewer. By studying the architecture upfront, creating collaborative plans, and drawing hard boundaries through fast computational sensors, I stepped up to orchestrate the behavior of the system.

"Vibe coding" gives me the speed of the Shippers, but my Harness Engineering workflow guarantees the quality that my identity as a Builder demands.
