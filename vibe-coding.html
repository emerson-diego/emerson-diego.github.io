How I Build Software with AI in 2026: From Vibe Coding to Harness Engineering

April 17, 2026

AI didn’t just speed up coding; it changed where engineering discipline must live. I've been diving headfirst into so-called 'intuitive programming' [ou 'vibe coding'] and the reality hit me hard: the execution barrier has plummeted. Today, it is perfectly feasible to build a clone of a complex system in a few days using modern generative coding agents. Code itself has become a commodity.

In this process, I realized that the mental model of treating AI as an 'intern' — reading every single line of code it generates — simply doesn't work in practice. It's not scalable. However, to avoid low-quality results, I fundamentally changed my mental model: I stopped focusing solely on the prompt and started designing a control system.

Let's be honest: comprehensive documentation and rigorous testing have always been critical, but they were often the first things neglected by small teams in the name of speed. Today, the game has changed. What used to be survivable sloppiness becomes catastrophic when an agent can generate thousands of lines in seconds. There is simply no excuse anymore for skipping robust tests and living documentation; they are the fundamental guardrails required to harness these tools effectively.

Below, I detail exactly how I manage my projects today. This is not just an abstract philosophy, but a replicable operational model that combines my personal approach to development with the practices of Harness Engineering and Specification-Driven Development. (For context, my daily workflow is currently powered by a mix of Cursor—running Opus or Sonnet—Codex, and Antigravity).

Step 1: The Foundation (I never start with code)

Before even kicking off the project, my golden rule is to study the product's architecture. AI agents operating without clear boundaries will try to guess the path, and that inevitably leads to hallucinations.

  • Living Documentation: I continuously use the project's documentation to guide the entire workflow. This documentation acts as my single source of truth.
  • The "Spec Pack": In traditional software engineering, we don't push anything to production without clear requirements. I do the exact same thing with AI. For every feature, I create a specification package detailing the exact constraints before execution. If the AI doesn't know exactly what is expected of it, my control system will have nothing to validate against.

Example: Spec Pack for an Auth Endpoint

  • Intent: Authenticate users via JWT and prevent brute-force attacks.
  • Constraints: Must use Redis for rate limiting; latency under 50ms.
  • Acceptance Criteria: Returns HTTP 429 when the limit is hit (5 attempts/min); includes 'Retry-After' header.
  • Contract (I/O): Input: {email, password}. Output: {token} or standard error block.
  • Edge Cases: Invalid email format (HTTP 400), correct credentials but locked account (HTTP 403).

Step 2: Collaborative Planning

I have always worked with and appreciated the spiral development model, focusing strictly on one feature at a time. However, the real game-changer happens in the moment just before implementation, by utilizing the planning tool integrated into whichever harness is currently in use.

  • Execution Plan: I never just show up and say, "create endpoint X." Before any execution, I explicitly write down the plan defining what needs to be done, how it should be done, the constraints, and how the outcome will be validated.
  • Space for AI: Before I let the AI generate any code, I ask it to analyze the plan and offer architectural improvements. This collaborative phase leverages the model's inferential capacity to anticipate issues before we burn tokens and time.
  • Feedforward Guides: Alongside the plan, I provide "Feedforward"—instructions that shape the agent's behavior before errors even happen (like project conventions and architectural templates). This drastically increases the chances of getting it right on the first try.

Step 3: Implementation and Fast Sensors (Keep Quality Left)

With the plan validated, the AI agent goes to work. However, my confidence doesn't stem from the fact that the AI is "smart," but rather from my network of automated Feedback Sensors.

I strictly follow the "Keep quality left" principle—meaning tests must run as early as possible in the cycle. To ensure my agent's repetition loop is incredibly fast, I rely on purely computational sensors (linting, type-checking, contract tests, and focused unit/integration tests). Fast sensors exist to shorten the correction cycle, not just to "guarantee quality".

However, if the AI is writing the code, who checks the AI? Mutation tests act as the ultimate "sensor of sensors." By actively modifying the code and expecting tests to fail, I ensure my test suite genuinely catches AI hallucinations and logic drift, rather than just passing vacuously. If the AI breaks a contract from Step 1, these sensors catch the error and the agent attempts to correct itself automatically.

Step 4: The "Control Loop" (How I handle AI failures)

Let's be clear: AI fails, and the worse the model, the more it will fail. When the error is simple, the AI reads the test logs and fixes it autonomously. But when the agent gets stuck in a loop, makes low-confidence decisions, or produces that dreaded scrambled output, the problem becomes mine.

That's where I step in as a true Harness Engineer. Instead of opening the file and simply patching the line of code the AI messed up, I close the Steering Loop:

  • If the spec was unclear, I improve the examples.
  • If the tests missed the bug, I add a new sensor.
  • If the architecture drifted, I update the feedforward rules.

I continuously evolve the control system so that this specific error never depends on my manual intervention again in the future.

A Real Failure Case: The Connection Leak

Recently, an AI agent generated an excellent data extraction routine but reliably 'hallucinated' skipping the closure of the database connection pool on failed retries. My fast sensors (specifically, a mock integration test assessing connection bounds) caught the leak immediately and broke the build. Instead of simply opening the file and patching the close() command myself, I closed the Steering Loop: I updated my Feedforward Guides with a strict architectural rule stating exactly how database sessions must be handled using context managers. The agent learned the fundamental boundary permanently.

Step 5: The Delivery Spiral and Client Feedback

Since I work in a spiral model, whenever an issue successfully passes through my network of automated sensors, I take the result directly to the client for rapid feedback. The client validates behavior and value; the sensors validate consistency and security.

If the client requests adjustments, the process immediately returns to my collaborative planning phase.

Only when the code is fully validated do I offload the heavy lift to the CI/CD pipeline for load and performance testing. In short: fast sensors protect iteration speed; slow sensors protect release confidence.

The Limits of Vibe Coding

While this workflow is incredibly powerful for scaffolding features rapidly, it has clear limits. The generative approach breaks down during delicate refactoring of legacy systems, when dealing with implicit undocumented business rules, or when resolving sensitive security architectures (managing secrets and broad permissions). In these areas, the risk is too high and human leverage remains absolute.

A Minimum Viable Harness

If you want to transition from "Shipper" to "Harness Engineer" today, you do not need an astronomically complex setup. You can start with a Minimum Viable Harness tomorrow:

  • Feature Spec: Write exactly what needs to be done and what absolutely cannot happen.
  • Execution Plan: Force the AI to output exactly how it will code the solution before it writes a single line.
  • Local Sensors: Enforce a linter, type-checker, and a test suite that runs automatically on every save.
  • Stopping Rule: If the AI hallucinates the exact same error three times, stop the loop. Take control.
  • Escalation Trigger: If a sensor fails and the AI cannot independently parse the log to fix it, it must escalate to you.

The Bottom Line

I stopped being an AI-generated code reviewer. By studying the architecture upfront, creating collaborative plans, and drawing hard boundaries through fast computational sensors, I stepped up to orchestrate the behavior of the system.

AI gives me leverage; harness engineering gives me control.
